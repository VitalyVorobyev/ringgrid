# Migrating from 0.10 to 0.11

Release 0.11 makes every target combination a one-call constructor, and stops
storing origin-dot coordinates. Two breaking changes need attention: **rect
lattice coordinates are now centered**, and **target JSON is schema v6**.

## Rect `grid_coord` is centered

Rect lattice coordinates used to be 0-based from the top-left corner
(`0..=23` on a 24-wide board). They are now **centered**, matching the
convention the hex axial lattice already used:

| Board | 0.10 range | 0.11 range |
|---|---|---|
| 24 × 24 | `0..=23` | `-11..=12` |
| `n` wide | `0..=n-1` | `-(n-1)/2 ..= n-1-(n-1)/2` |

Cell `(0, 0)` is now the **central** cell rather than a corner. The physical
target is unchanged: board millimeters, printed geometry, and `board_xy_mm` are
all identical — only the integer labels moved.

**What to update**

- Anything comparing `grid_coord` against stored values: re-derive or shift by
  `(cols-1)/2` and `(rows-1)/2`.
- Ground-truth files keyed by rect cell coordinate — regenerate them.
  `tools/gen_synth_rect.py` emits the new convention.
- Code that treated `grid_coord == [0, 0]` as "the board corner". The corner is
  now `[-(cols-1)/2, -(rows-1)/2]`; `[0, 0]` is the center.

Hex targets are unaffected — their coordinates were already centered.

## Origin fiducials no longer store coordinates

`OriginFiducials` carried `dot_radius_mm` **and** `dots_mm`, a list of absolute
board-millimeter dot centers. Absolute coordinates go stale: change a board's
rows, columns, or pitch and the stored triad silently describes points that no
longer sit in the lattice gaps — or lie off the marker field entirely, where the
anchoring homography has to extrapolate to reach them. Nothing rejected that.

Positions are now **derived** from the lattice — an L of three lattice gaps
around cell `(0, 0)` — and only the size is stored:

```rust
// 0.10
OriginFiducials { dot_radius_mm: 1.4, dots_mm: vec![[161.0, 161.0], /* … */] }

// 0.11
OriginFiducials { dot_radius_mm: 1.4 }
```

Read positions back with `TargetLayout::fiducial_dots_mm()` (Rust),
`TargetLayout.fiducial_dots_mm()` (Python), or `target_fiducial_dots_mm(json)`
(WASM). Prefer the constructors, which derive the size too:

```rust
TargetLayout::plain_rect(14.0, 24, 24, 5.6, 2.8, OriginDots::Auto)?
```

The dev CLI's `--dot_mm x,y` flag is removed; use `--dots` (with an optional
`--dot_radius_mm` to override the derived size).

## Target JSON is schema v6

`ringgrid.target.v6` differs from v5 only in `fiducials`, which now carries just
`dot_radius_mm`. Writers emit v6; loaders still accept v5 and v4.

**v5 files load with one guard.** The stored `dots_mm` are *compared against* the
derived positions rather than ignored, and a mismatch is an error
(`LegacyDotsMismatch`) — a file whose dots are printed somewhere this build
would not look for them cannot silently detect against the wrong place.

In practice only **rect** targets generated with `fiducials = "auto"` by
0.10.x are affected: their third dot sat one pitch away from where 0.11 derives
it. Regenerate the target and re-print it. Hex targets, the `rect_24x24` preset,
and every hand-authored L migrate silently — including physical boards printed
from `rect_24x24`, whose triad the derived placement reproduces exactly.

## Coded targets with fiducials are rejected

`TargetLayout::new` now returns `TargetValidationError::CodedWithFiducials` for
coded targets carrying origin dots. This was already documented as the one
excluded combination and already rejected by the CLI recipe; the rule now lives
in one place and applies to every construction path, including the JSON loaders.
`RecipeError::CodedWithFiducials` is removed — it surfaces as
`RecipeError::Validation`.

## New constructors (additive)

Every row of the target matrix is one call, in Rust and Python:

| Combination | Rust | Python |
|---|---|---|
| hex + coded | `TargetLayout::coded_hex(…)` | `TargetLayout.coded_hex(…)` |
| rect + coded | `TargetLayout::coded_rect(…)` | `TargetLayout.coded_rect(…)` |
| hex + plain | `TargetLayout::plain_hex(…, OriginDots)` | `TargetLayout.plain_hex(…, dots=)` |
| rect + plain | `TargetLayout::plain_rect(…, OriginDots)` | `TargetLayout.plain_rect(…, dots=)` |

`TargetLayout::with_name(...)` replaces the derived geometry-based name.
